# Анализ: Проблемы крипто-агента и качество торговых решений

## Контекст

Исследованы все модули крипто-торговли: bybit-client.ts, monitor.ts, snapshot.ts, trade.ts, state.ts, config.ts, indicators.ts, risk.ts, workspace (SOUL.md, AGENTS.md, HEARTBEAT.md, TOOLS.md), скрипты (crypto_check.sh, crypto_cron.sh). Цель — выявить причины низкого качества торговых решений и предложить путь к 70%+ winrate при минимум 2 сделках в день.

## Данные и наблюдения

### 1. Текущая логика принятия решений (monitor.ts:205-281)

Функция `analyzePair()` — единственное место, где генерируются сигналы. Анализ:

**Используется:**

- H4 (240) — определение тренда через EMA50 vs EMA200
- M15 (15) — RSI для входа + ATR для стоп-лосса
- Funding rate — фильтр (блокирует вход при extreme funding)
- Support/Resistance — простой min/max за 20 баров

**Логика входа (LONG):**

```
trendBias === 'BULLISH' && priceVsEma === 'ABOVE'
  && (rsi15m < 40 || distToSupport < 1.5%)
```

**Логика входа (SHORT):**

```
trendBias === 'BEARISH' && priceVsEma === 'BELOW'
  && (rsi15m > 60 || distToResistance < 1.5%)
```

### 2. Выявленные проблемы

#### КРИТИЧЕСКАЯ #1: Нет системы confluence scoring

Текущий подход — бинарный: условия совпали → вход. Нет оценки КАЧЕСТВА сигнала. Профессиональные трейдеры используют систему оценки от -100 до +100, где вход разрешён только при score > 70. Текущая система эквивалентна "score > 0" — любой слабый сигнал проходит.

**Влияние:** Входы на слабых сигналах = 50/50 winrate в лучшем случае.

#### КРИТИЧЕСКАЯ #2: Отсутствие анализа объёма

Volume полностью игнорируется при принятии решений. OHLC содержит `volume` и `turnover`, но `buildMarketAnalysis()` не включает их в анализ. Breakout на низком объёме — ловушка. Профессиональные трейдеры подтверждают каждый сигнал объёмом.

**Влияние:** Ложные breakout'ы, входы в ловушки, low-quality setups.

#### КРИТИЧЕСКАЯ #3: Примитивный Support/Resistance

`calculateSupportResistance()` — просто `Math.min(lows)` и `Math.max(highs)` за 20 баров. Это не уровни, а диапазон. Реальные уровни — кластеры цен где было много отбоев, pivot points, профили объёма.

**Влияние:** SL/TP ставятся на неадекватных уровнях → выбивает по стопу.

#### КРИТИЧЕСКАЯ #4: Нет multi-timeframe confluence

Используются 2 TF (H4 + M15), но без реальной конфлюэнции. H4 определяет тренд — и всё. Нет проверки что M15 подтверждает H4, нет анализа H1 (промежуточный TF), нет M5 для точного входа. Config имеет `zonesTF: '60'` и `precisionTF: '5'`, но они НЕ ИСПОЛЬЗУЮТСЯ в коде.

**Влияние:** Входы не синхронизированы между таймфреймами → ранние входы → стопы.

#### КРИТИЧЕСКАЯ #5: Нет детекции рыночного режима

Нет определения: trending vs ranging vs volatile market. RSI в ranging-рынке и trending-рынке работает СОВЕРШЕННО по-разному. В тренде RSI > 60 — нормально, в range — перекупленность. ATR как абсолютное значение не показывает "волатильность выше/ниже нормы".

**Влияние:** Одна и та же стратегия применяется ко всем режимам → провалы в ranging/choppy рынке.

#### КРИТИЧЕСКАЯ #6: Нет orderbook/liquidation analysis

Bybit API предоставляет:

- Orderbook depth (до 1000 уровней) — видеть стены покупок/продаж
- Liquidation data (WebSocket) — зоны принудительных ликвидаций
- Open Interest history — расхождение OI и цены → мощный сигнал

Текущий код берёт только 1 точку OI для отображения. Никакой аналитики.

**Влияние:** Упущенные мощные сигналы (OI divergence, liquidation cascades).

#### ВАЖНАЯ #7: snapshot.ts vs monitor.ts дублирование

snapshot.ts собирает данные для AI-агента (OpenClaw), monitor.ts — для автоматической торговли. Но AI-агент получает те же данные что и monitor, без дополнительного контекста (orderbook, volume profile, OI history, funding history). AI-агент не может принять лучшее решение чем автомат, если данных столько же.

#### ВАЖНАЯ #8: RSI расчёт неоптимален

`calculateRsi()` использует simple average вместо smoothed/exponential moving average (стандарт Wilder). Результат отличается от TradingView и других платформ. При period=14 расхождение может быть 2-5 пунктов — критично на границах зон (68 vs 72).

#### ВАЖНАЯ #9: Market order по умолчанию

`executeSignals()` отправляет Market ордера (строка 330). SOUL.md говорит "prefer limit orders", но код это игнорирует. Market orders = проскальзывание + комиссия maker vs taker.

**Влияние:** Потеря 0.03-0.06% на каждой сделке от проскальзывания.

#### СРЕДНЯЯ #10: Нет фильтра времени

Нет учёта времени суток. Азиатская сессия (00-08 UTC) — низкая ликвидность, US Open (13:30 UTC) — высокая волатильность, Sunday evening — gaps. FOMC/CPI фильтр есть в SOUL.md как правило для AI, но не в коде.

#### СРЕДНЯЯ #11: Нет фильтра корреляции

12 пар торгуются независимо. Но BTC, ETH, SOL часто двигаются вместе. 3 лонга на коррелированных активах = 1 большая позиция, а не 3 диверсифицированных. Max 3 позиции не спасает от корреляционного риска.

#### СРЕДНЯЯ #12: Нет истории и обратной связи

Нет анализа прошлых сделок (backtest feedback). State хранит daily stats, но не анализирует паттерны: "какие сигналы приводят к win, какие к loss". Нет обучения на ошибках.

## Архитектурная проблема: Два пути принятия решений

Сейчас есть конфликт:

1. **monitor.ts** — автоматический трейдер с жёсткой логикой (analyzePair)
2. **AI-агент (OpenClaw crypto-trader)** — получает snapshot.ts данные и решает сам

Проблема: monitor.ts слишком примитивен, а AI-агент ограничен теми же данными. Нужен **один путь**: собрать МАКСИМУМ данных → подготовить КАЧЕСТВЕННЫЙ snapshot → AI-агент принимает решение с confluence scoring.

## Выводы

1. **Качество данных**: snapshot собирает ~60% нужных данных. Отсутствуют: orderbook depth, volume analysis, OI history, funding history, liquidation zones, корреляция пар
2. **Качество анализа**: Confluence scoring отсутствует. Бинарная логика "сигнал есть / нет" не обеспечивает 70% winrate
3. **Качество входов**: Market orders, примитивные S/R, нет multi-TF confirmation
4. **Качество risk management**: Хорошее — riskPerTrade 2%, maxDailyLoss, killswitch, partial close, trailing SL. Это сильная сторона
5. **Архитектура**: Конфликт monitor.ts (автомат) vs AI-агент. Нужна унификация

## Токен-экономика

Бюджет: **$3/день** через OpenRouter на крипто-трейдера.

Текущий подход: LLM (GPT-5.2 через OpenClaw) принимает ВСЕ торговые решения. Heartbeat каждые 2 часа = 12 вызовов × ~7500 токенов = ~90k токенов/день.

**Стоимость по моделям:**
| Модель | $/вызов | $/день (12 calls) | % от бюджета |
|--------|---------|-------------------|-------------|
| GPT-5.2 | ~$0.10 | ~$1.20 | 40% |
| GPT-4o | ~$0.04 | ~$0.48 | 16% |
| GPT-4o-mini | ~$0.005 | ~$0.06 | 2% |

**Вывод:** Бюджет комфортный. GPT-4o как основная модель ($0.48/день = 16% бюджета). Но архитектура "КОД готовит, LLM решает" нужна для КАЧЕСТВА, а не экономии:
- Confluence scoring в TypeScript → LLM получает pre-computed данные → лучше решения
- Auto-trade fallback при недоступности OpenRouter
- Хранение истории в JSON файлах (не Redis — overkill)

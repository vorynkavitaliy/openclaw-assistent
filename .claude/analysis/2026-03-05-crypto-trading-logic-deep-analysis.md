# Глубокий анализ торговой логики крипто-трейдера

## 1. Конфигурация и параметры риска (config.ts)

### Параметры риска

| Параметр | Значение | Оценка |
|----------|----------|--------|
| `riskPerTrade` | 2% от баланса | Норма |
| `maxRiskPerTrade` | $250 | Жёсткий кеп |
| `maxDailyLoss` | $500 | Умеренно |
| `maxStopsPerDay` | 2 | Очень консервативно |
| `maxOpenPositions` | 3 | Норма |
| `defaultLeverage` | 3x | Консервативно для крипто |
| `maxLeverage` | 5x | Разумный потолок |
| `minRR` | 2.0 | Правильно |

### Проблемные параметры

**maxStopsPerDay = 2 — критически консервативно.**
При 2 стопах подряд система встаёт до следующего дня. На крипто рынке 2 стопа в рамках нормальной торговли вполне вероятны (win rate 50-55% = 2-3 стопа из 5 попыток). Это создаёт bottleneck: хотим 2-4 сделки в день, но 2 стопа = финиш.

**maxDailyLoss = $500, но aggregate risk limit = $250 (50% от maxDailyLoss).**
Если открыта 1 позиция с риском $250 — новые позиции уже заблокированы агрегатным лимитом. Де-факто не 3 позиции одновременно, а 1-2.

**riskPerTrade = 2% при maxRiskPerTrade = $250** — при балансе >$12,500 кеп $250 начнёт срабатывать и отсекать расчётный риск.

**ecosystemGroups** — 9 из 12 пар покрыты группами (3 группы по 3 пары). Остаются вне групп: BTCUSDT, XRPUSDT, DOGEUSDT. Это означает, что если BTC занят, можно торговать только 3 некоррелированных пары вместо 12.

**staleOrderMinutes = 30** — лимитные ордера живут 30 минут. Это адекватно.

**atrSlMultiplier = 1.5** — умеренно. На M15 ATR для BTC часто 150-300 USD, SL = 225-450 USD от entry. При minRR=2 нужно движение 450-900 USD. Реалистично, но не каждый день.

**Вывод по конфигу:** система настроена скорее под "качество, а не количество". Параметры не агрессивные. Главный риск — что при maxStopsPerDay=2 система может остановиться к середине дня даже при нормальной торговле.

## 2. Confluence Scoring (confluence.ts)

### Веса модулей

| Модуль | Вес | Диапазон scores |
|--------|-----|-----------------|
| Trend | 25% | -10..+10 |
| Momentum | 15% | -10..+10 |
| Volume | 15% | -10..+10 |
| Structure | 15% | -10..+10 |
| Orderflow | 15% | -10..+10 |
| Regime | 15% | -10..+10 (фактически -10..+8) |

### Формула

```
raw = trend*0.25 + momentum*0.15 + volume*0.15 + structure*0.15 + orderflow*0.15 + regime*0.15
total = round(raw * 10) + conflictPenalty
```

Максимум теоретический: `(10*0.25 + 10*0.15*5) * 10 = (2.5 + 7.5) * 10 = 100`. Верно.

### Реальный диапазон scores при типичных рыночных условиях

**Оценка типичного рынка (не экстремального):**

- Trend: чаще всего D1+H1 не в полном alignment → score 4-7 (не 10)
- Momentum: RSI в нейтральной зоне 40-60, MACD без явного сигнала → score 1-4
- Volume: relativeVolume обычно 0.8-1.3x → score -1 до +1
- Structure: цена редко у самого support/resistance → score 0-3
- Orderflow: OI FLAT, funding NEUTRAL → score 0-2
- Regime: чаще WEAK_TREND или RANGING → score 2-4

**Типичный позитивный сигнал:**
`(6*0.25 + 3*0.15 + 1*0.15 + 2*0.15 + 1*0.15 + 4*0.15) * 10 = (1.5 + 0.45 + 0.15 + 0.3 + 0.15 + 0.6) * 10 = 3.15 * 10 = 31-35`

**Хороший сигнал (сильный тренд):**
`(8*0.25 + 6*0.15 + 3*0.15 + 4*0.15 + 3*0.15 + 8*0.15) * 10 = (2.0 + 0.9 + 0.45 + 0.6 + 0.45 + 1.2) * 10 = 56`

**Сильный сигнал:**
`(10*0.25 + 8*0.15 + 5*0.15 + 5*0.15 + 5*0.15 + 8*0.15) * 10 = (2.5 + 1.2 + 0.75 + 0.75 + 0.75 + 1.2) * 10 = 72`

**Реальный диапазон типичных сигналов: 25-55 по модулю.**

### Потенциальные проблемы

1. **Volume score полагается на volumeDelta**: `Math.min(5, Math.round(Math.abs(delta) / 10000))` — дельта 50,000 даёт score +5. Но для DOGE/MATIC дельта может быть значительно меньше, что снижает этот компонент до 0-1 даже при хорошем объёме.

2. **Structure score слишком узкий порог:** `distToSupport < 0.5%` для +5 баллов. BTC двигается 0.5% за минуту — этот порог почти никогда не будет выполнен в момент анализа.

3. **Conflict penalty -15** срабатывает при trend>5 и momentum<-5. Если тренд bullish (score 7+) но RSI перегрет (сигнализирует разворот) — итоговый score может упасть ниже порога, что по факту правильно.

4. **Regime weight 15% с максимумом +8** (STRONG_TREND) — асимметрия: CHOPPY даёт -10, но STRONG_TREND только +8. Это слегка занижает оценку в трендовых условиях.

## 3. Regime Detection и пороги (regime.ts)

### Алгоритм

Score-based: trendScore (0-6) + volatilityScore (-3 до +5).

| Regime | Условие | Порог confluence |
|--------|---------|-----------------|
| STRONG_TREND | trendScore >= 5 | 30 |
| WEAK_TREND | trendScore >= 3 | 45 |
| RANGING | default | 50 |
| VOLATILE | volatilityScore >= 4 | 65 |
| CHOPPY | crossCount >= 4 AND trendScore <= 1 | 80 |

### Проблема несоответствия порогов и реальных scores

**RANGING (порог 50):** типичный score 31-35. Большинство сигналов в боковике не пройдут порог 50. Фактически RANGING = "не торгуем".

**VOLATILE (порог 65):** для score 65 нужен почти идеальный набор — тренд 9+, момент 7+, хороший объём. При высокой волатильности рынок хаотичен, и собрать такой score крайне сложно. Порог 65 при VOLATILE = де-факто блокировка.

**CHOPPY (порог 80):** score 80 нереален при чопповом рынке. Это правильно — в choppy не торгуем.

**Вывод по порогам:** тройное несоответствие: RANGING=50 и VOLATILE=65 создают "мёртвые зоны" где сигналы технически появляются (score 35-50) но не проходят порог. Система будет работать эффективно только при STRONG_TREND (порог 30) и WEAK_TREND (порог 45).

## 4. LLM Advisor (llm-advisor.ts)

### Модель и параметры
- Модель: `anthropic/claude-sonnet-4` (через OpenRouter)
- Temperature: 0.1 (детерминированный режим — правильно)
- max_tokens: 1024
- Таймаут: 30 сек

### Контекст дня
Хорошо проработан: баланс, открытые позиции, дневной P&L, wins/losses, urgency (HIGH/MODERATE/NORMAL на основе оставшихся часов и нужных сделок).

### Данные на вход для каждого сигнала
- Direction, Entry, SL, TP, R:R
- Confluence total score + signal type + confidence %
- Режим рынка
- История confluence scores (последние 4 цикла)
- Топ-4 детальных фактора

### Проблемы промпта

1. **Смещение к ENTER**: fallback при ошибке парсинга JSON — ENTER. При сбое LLM — тоже ENTER. Это создаёт bias: любая техническая проблема = вход в сделку.

2. **SYSTEM_PROMPT противоречив:** "Default choice for signals with confluence > 40 and confidence > 60%" — но сигналы с score 30-45 (WEAK_TREND/STRONG_TREND) уже прошли порог и тоже попадают к LLM. LLM может их пропустить, руководствуясь правилом ">40".

3. **Urgency логика агрессивна:** при hoursLeft <= 8 и нехватке сделок — приоритет ENTER. Это создаёт дневную концентрацию рисков в конце торговой сессии.

4. **Score history** — передаётся только список чисел без таймфреймов. LLM не знает, сколько времени прошло между cycles (10 минут). История из 4 точек = 40 минут — мало для понимания тренда.

5. **Нет данных о новостях** — ни экономический календарь, ни crypto news не передаются. LLM принимает решения только по техническому контексту.

6. **WAIT добавляет в watchlist** — но логика watchlist и как это влияет на следующие cycles не видна в промпте.

### Стоимость
При 12 парах * ~600 токенов промпт ≈ 7200 токенов prompt + ~200 completion = ~$0.023 за цикл. При 144 cycles/день ≈ $3.3/день. Приемлемо.

## 5. Signal Executor (signal-executor.ts)

### Фильтры после LLM (в порядке применения)

1. `state.canTrade()` — kill switch, stop day, mode != execute, max positions, aggregate risk
2. `existing position` — уже есть позиция по этой паре → SKIP
3. `pending order` — уже есть ожидающий лимитный ордер → SKIP
4. `ecosystem filter` — группа экосистемы уже занята → SKIP
5. `calcPositionSize` — qty = 0 (balance = 0) → SKIP
6. `maxRiskPerTrade` — риск > $250 → SKIP
7. `insufficient margin` — margin > available → SKIP

### Сколько сигналов реально проходит

Допустим, из 12 пар к LLM дошли 3 сигнала (STRONG_TREND), LLM сказал ENTER по 2:

- Если 1 позиция уже открыта → aggregate risk $250 ≥ maxTotalRisk $250 → новые заблокированы
- Если BTC занят, ETHUSDT и ARBUSDT в одной группе → ECOSYSTEM_OCCUPIED по ARBUSDT
- При defaultLeverage=3 и balance=$5000: requiredMargin = (95000 * 0.001) / 3 = ~$32 — обычно проходит

**Реальный bottleneck executor'а — aggregate risk limit.** При riskPerTrade=2% и balance=$10,000: riskAmount=$200. maxTotalRisk = $500 * 0.5 = $250. То есть можно открыть только 1 позицию с риском $200, а следующая добавит суммарный риск $400 → превысит $250 → BLOCKED.

**При стандартном балансе ($5,000-$15,000) система де-факто ограничена 1 позицией одновременно**, а не 3.

## 6. Market Analyzer — Pre-filters (market-analyzer.ts)

### Pre-filters до confluence scoring

1. **Пустой orderbook** → skip (rare, но бывает при API issues)
2. **Spread > 0.1%** → skip. Для топ-пар (BTC, ETH) spread обычно 0.001-0.02% → проходят все. Для DOGE/MATIC может быть 0.05-0.08% в пики → иногда режется.
3. **Funding rate > 0.1% или < -0.1%** → skip. Bybit funding обновляется каждые 8 часов. Обычный диапазон 0.01-0.05% → проходит. Но при бычьих ралли BTC funding может достигать 0.1-0.3% → BTCUSDT будет заблокирован.
4. **volumeProfile = null** (пустые M15 candles) → skip (API error scenario)
5. **ATR = 0 или price = 0** → skip
6. **Entry deviation > 5% от lastPrice** → skip (bid1/ask1 sanity check)
7. **SL distance < 0.1% или > 20%** → skip

### Что реально режется

- Funding filter — самый активный: в bull run периоды BTC и ETH часто имеют funding 0.08-0.12%, что близко к порогу 0.1% (0.001 в абсолютных единицах).
- SL sanity check — может срабатывать для стейблкоинов или если ATR аномально мал.

**Ключевая проблема:** funding threshold 0.001 (0.1%) — это стандартный bybit funding. Если funding = 0.0008 (0.08%) — проходит. Если 0.0012 (0.12%) — нет. На практике BTC funding во время трендов регулярно бывает 0.05-0.15%, создавая непостоянство фильтра.

## 7. Общая оценка: bottleneck анализ

### Воронка сигналов (из 12 пар, 1 цикл = 10 минут)

```
12 пар
  ↓ funding/spread pre-filter (-2 в среднем в bull run)
10 пар проходят pre-filters
  ↓ confluence < threshold (RANGING=50, WEAK_TREND=45)
2-3 пары достигают порога (при STRONG_TREND больше)
  ↓ LLM (температура 0.1, склонность к ENTER)
1-2 пары → ENTER
  ↓ executor: aggregate risk, ecosystem, existing positions
0-1 реальный ордер за цикл
```

### Реалистичный прогноз сделок в день

- Циклов в день: 144 (каждые 10 минут)
- Цикл с сигналом (предположительно): ~10-15% циклов → 14-21 потенциальных сигналов
- После LLM фильтра → 8-15 ENTER решений
- После executor (aggregate risk, ecosystem, positions) → 2-5 реальных ордеров
- Из них исполняется (лимитный ордер, 30 мин) → 1-3 фактических входа

**Цель 2-4 сделки в день ДОСТИЖИМА, но в узком диапазоне условий:**
- Рынок в STRONG_TREND или WEAK_TREND (не RANGING/VOLATILE)
- Funding rates в норме
- Стопы не срабатывают (иначе после 2 стопов = конец дня)
- Агрегатный риск не блокирует через 1-ую позицию

## Ключевые рекомендации

### Приоритет 1 (блокируют цель 2-4 сделки)

1. **maxStopsPerDay: 2 → 3-4.** При цели 2-4 сделки в день и любом реальном winrate 50-60% ограничение в 2 стопа слишком жёсткое.

2. **Aggregate risk limit: 50% maxDailyLoss = $250.** При riskPerTrade=$200 (2% от $10k) одна позиция занимает 80% лимита. Нужно поднять до 70-80% maxDailyLoss или убрать это ограничение, оставив только maxOpenPositions.

3. **Порог RANGING: 50 → 42.** При типичных scores 31-45 система не торгует в боковике вообще. Снижение до 42 откроет доступ к боковиковым сетапам у уровней S/R.

### Приоритет 2 (улучшают качество)

4. **Structure threshold: 0.5% → 1.0%.** Порог 0.5% слишком узкий, +5 баллов структуры почти недостижимы. При 1.0% появится значимый вклад этого модуля.

5. **LLM fallback при parse error → SKIP вместо ENTER.** Текущий fallback-to-ENTER при ошибках создаёт неконтролируемые входы без настоящего LLM решения.

6. **Score history с таймстемпами.** Передавать `{score, time}` вместо просто чисел — LLM сможет оценить скорость изменения confluence.

### Приоритет 3 (долгосрочно)

7. **Volume delta нормализация по паре.** Разные пары имеют разные абсолютные volume delta. Нормализовать как % от среднего volume за последние N свечей.

8. **Добавить время дня в urgency логику.** Азиатская сессия (00-08 UTC) менее активна — urgency HIGH в это время провоцирует входы при низком volume.

9. **Мониторинг реального distribution confluence scores.** Логировать scores всех пар, не только те что прошли threshold, для калибровки порогов на реальных данных.

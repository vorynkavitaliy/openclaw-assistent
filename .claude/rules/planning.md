# Правила планирования

Когда пользователь просит **спланировать** что-либо (фичу, рефакторинг, миграцию, задачу):

## Обязательный процесс

1. Изучить текущую кодовую базу и контекст
2. Декомпозировать задачу на конкретные этапы
3. Сохранить план в `.claude/planning/YYYY-MM-DD-<slug>.md`
4. Формат имени файла: дата + краткое описание через дефис (например `2026-03-03-add-trailing-stop.md`)

## Шаблон файла плана

```markdown
---
type: plan
title: <название>
date: YYYY-MM-DD
status: draft | approved | in-progress | done
priority: high | medium | low
complexity: S | M | L | XL
---

# План: <название>

## Цель
<что нужно сделать и зачем>

## Анализ текущего состояния
<что есть сейчас, какие модули затронуты>

## Затронутые файлы
- `src/path/file.ts` — <что изменится>

## Этапы реализации

### 1. <Название этапа> (агент: developer/tester/analyst)
- Описание действий
- Ожидаемый результат
- Оценка: S/M/L/XL

### 2. <Следующий этап>
...

## Риски
| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| ... | low/med/high | low/med/high | ... |

## Definition of Done
- [ ] Код написан
- [ ] Тесты написаны и проходят
- [ ] `npm run lint && npm run build` — успешно
- [ ] Code review пройден
```

## Сложность

| Размер | Время | Когда использовать |
|--------|-------|--------------------|
| S | < 1 час | Одна функция, мелкий фикс |
| M | 1-4 часа | Новый модуль, несколько файлов |
| L | 4-8 часов | Крупная фича, затрагивает много модулей |
| XL | > 1 день | Архитектурные изменения, миграции |

## Правила

- План — конкретный: файлы, функции, типы — не абстракции
- Каждый этап привязан к агенту (developer, tester, analyst)
- После developer — обязательно tester
- Для торговых модулей — обязательно оценка рисков (деньги!)
- План должен быть реализуемым — не описывай то, что невозможно проверить
- После планирования — кратко сообщить пользователю структуру плана и путь к файлу
